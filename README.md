# Agents Advanced Course

This repository is a set of small, focused demos for building AI agents with two different approaches:

- building agent loops directly in Python
- building agents with the Google ADK
- working with artifacts generated by agents
- adding blocking and non-blocking human approval flows

The code is organized as independent demo folders rather than a single application. Each folder is meant to illustrate one agent pattern clearly and with minimal surrounding infrastructure.

## Repository Structure

| Folder | What it demonstrates |
| --- | --- |
| `bulding_agent_from_scratch/` | Basic agent loops, tool use, and memory without ADK |
| `bulding_agent_adk/` | A multi-agent SQL assistant built with Google ADK |
| `artifacts_managment/` | An ADK paper-writing agent that creates a PDF artifact |
| `human_in_the_loop/` | Blocking and non-blocking approval workflows |

Notes:

- Several folder names use the original repo spelling, such as `bulding_*` and `artifacts_managment`.
- Each demo has its own `pyproject.toml`, so dependencies are scoped per folder.

## What You Will Learn

Across the demos, the repo covers these recurring agent ideas:

- perception, decision, and action loops
- tool calling from an LLM
- simple memory/state accumulation
- multi-agent orchestration
- callback hooks
- artifact generation and persistence
- resumable workflows
- human approval as part of tool execution

## Prerequisites

- Python 3.11+
- `uv` recommended for dependency management and running scripts
- a Gemini-compatible API key for demos that call Google models
- for the PDF demo: a working Pandoc installation available to `pypandoc`

Most demos load environment variables from a `.env` file. At minimum, you will usually need:

```dotenv
GEMINI_API_KEY=your_key_here
DEFAULT_MODEL=gemini-3-pro-preview
```

Some demos use `GOOGLE_API_KEY` instead of `GEMINI_API_KEY`, depending on the SDK path used by the sample.

## Demo 1: Building Agent Loops From Scratch

Location: `bulding_agent_from_scratch/`

This folder shows how to build an agent loop manually without using ADK abstractions. It is the best place to start if you want to understand the mechanics before moving to framework-level concepts.

### Files

- `simple_agent.py`: introduces a base `Agent` class, a rule-based `BasicAgent`, and an `LLMAgent` that can call Python tools
- `loop_agent.py`: adds iterative reasoning, memory, explicit tool execution, and loop termination criteria
- `memory.py`: a lightweight sectioned memory store built with Pydantic

### Concepts Demonstrated

- the classic `perceive -> decide -> act` loop
- abstract base classes for agent behavior
- a rule-based baseline versus an LLM-backed agent
- tool registration by passing Python callables to the model config
- stateful memory that stores user questions, tool results, and model output
- custom termination logic

### How the Demo Works

#### `simple_agent.py`

This file contains two progressively more capable agents:

- `BasicAgent` handles a few hard-coded intents like weather and time
- `LLMAgent` wraps `modihub.llm.LLM` and forwards tools to the Gemini model

The example tool is `get_the_weather(location)`. The active entrypoint instantiates `LLMAgent` with:

- model: `models/gemini-pro-latest`
- system instruction: output responses in XML
- one callable weather tool

This demonstrates the smallest useful example of an LLM agent that can invoke a local function.

Run it with:

```bash
cd /Users/haruiz/open-source/agents_advanced_course/bulding_agent_from_scratch
uv run python simple_agent.py
```

#### `loop_agent.py`

This is the more interesting demo in the folder. `LoopAgent` extends the base agent and adds:

- a `Memory` object attached to internal state
- a Gemini client created with `google.genai`
- tool execution routed through `_execute_tool`
- iterative execution up to `max_iterations`
- a termination callback that stops when the model says `done`

The included tools are:

- `get_current_weather(location)`
- `get_current_time(location)`

The entrypoint seeds the agent with:

- an initial prompt asking for time and weather in New York
- tool access to the two functions above
- a system prompt instructing the agent to use tools and respond in XML

Run it with:

```bash
cd /Users/haruiz/open-source/agents_advanced_course/bulding_agent_from_scratch
uv run python loop_agent.py
```

#### `memory.py`

This file is not a runnable agent demo by itself, but it explains the memory model used by `loop_agent.py`. Memory is split into named sections, each containing one or more text entries. Converting the memory to a string produces the prompt context sent to the model.

That is the key teaching point: even very simple structured memory can be enough to create a useful looped agent.

## Demo 2: SQL Assistant With Google ADK

Location: `bulding_agent_adk/`

This demo shows how to move from hand-rolled loops to a structured multi-agent pipeline using the Google Agent Development Kit.

### Files

- `sql_assistant_agent/agent.py`: defines the agent graph and runtime entrypoint
- `tools.py`: database schema inspection and SQL execution tools
- `download_db.sh`: fetches a sample Northwind SQLite database
- `call_agent.http`: likely intended for manual API testing once the agent is served

### Concepts Demonstrated

- `LlmAgent`, `Agent`, and `LoopAgent` composition
- session state shared across agents
- before/after callbacks
- schema-aware SQL generation
- validation before execution
- escalation to terminate a loop
- artifact creation from tool output

### How the Demo Works

The system is composed of three agents:

1. `sql_junior_writer_agent`
   Generates a SQL query from the user prompt and stores it in session state under `sql_query`.
2. `sql_senior_writer_agent`
   Reviews the SQL, executes it if valid, and writes structured output to `sql_results`.
3. `CheckStatusAndEscalate`
   Stops the loop once the SQL result is marked valid.

These are wrapped by `root_agent`, a `LoopAgent` that retries the workflow for up to five iterations.

### Important Implementation Details

- `on_before_agent_call` loads the DB schema once and stores it in session state as `db_schema`
- `get_db_schema()` reads the SQLite schema from the database configured by `DB_PATH`
- `execute_sql_query()` runs the query, formats results as a Markdown table, and saves a CSV artifact named `sql_results.csv`
- the default prompt in `__main__` asks for the top 10 customers by total order amount

### Setup

Create a `.env` file in `bulding_agent_adk/` with values similar to:

```dotenv
GOOGLE_API_KEY=your_key_here
DB_PATH=/absolute/path/to/northwind.db
DEFAULT_MODEL=gemini-3-pro-preview
```

Download the sample database:

```bash
cd /Users/haruiz/open-source/agents_advanced_course/bulding_agent_adk
bash download_db.sh
```

If you save it in the project directory, `DB_PATH` can point to that local `northwind.db`.

### Run Options

Run the Python entrypoint directly:

```bash
cd /Users/haruiz/open-source/agents_advanced_course/bulding_agent_adk
uv run python sql_assistant_agent/agent.py
```

Or run it through ADK tooling:

```bash
cd /Users/haruiz/open-source/agents_advanced_course/bulding_agent_adk
uv run adk web --reload_agents
```

This demo is useful if you want to see how a framework can separate responsibilities across agents while preserving shared state.

## Demo 3: Artifact Management With a Paper Writer Agent

Location: `artifacts_managment/`

This demo focuses on agent outputs that are more than just text. The final result is a generated PDF saved as an artifact.

### Files

- `paper_writer_agent/agent.py`: defines the search, research, and PDF generation flow

### Concepts Demonstrated

- nested multi-agent composition
- structured outputs with Pydantic schemas
- using `AgentTool` to call one agent from another
- creating binary artifacts from generated content
- converting model output into a downloadable file

### How the Demo Works

The paper-writing pipeline has three stages:

1. `search_agent`
   Collects relevant background information. The code mentions a Google Search tool, but it is currently commented out, so in the current repo state it falls back to model knowledge unless you extend it.
2. `student_agent`
   Produces a structured academic paper using the schema `PaperDocumentStructure`, with sections for abstract, introduction, methodology, results, discussion, and conclusion.
3. `create_pdf`
   Converts the generated section content into Markdown, runs Pandoc through `pypandoc`, and stores the resulting PDF as an artifact.

`postdoc_agent` is a `LoopAgent` that coordinates the search and writing steps, and `root_agent` calls that workflow plus the PDF tool.

### Important Implementation Details

- `student_agent` writes its output to `paper_content`
- `create_pdf()` reads `paper_content` from tool context state
- the PDF is written to the provided filename, then reloaded and stored through `tool_context.save_artifact(...)`
- the default model is `gemini-flash-latest` unless `DEFAULT_MODEL` is set

### Setup Notes

This demo depends on Pandoc being installed on your machine. `pypandoc` is just the Python wrapper; the actual converter binary still needs to be present.

Run it from an ADK-compatible environment after installing dependencies in the folder:

```bash
cd /Users/haruiz/open-source/agents_advanced_course/artifacts_managment
uv sync
```

If you want to expose it through ADK’s UI, run the standard ADK web or server command from this directory.

### What to Look For

This demo is less about advanced prompting and more about output handling:

- turning structured content into a real file
- passing data between agents through state
- persisting non-text outputs as artifacts

## Demo 4: Human In The Loop

Location: `human_in_the_loop/`

This folder contains two approval patterns:

- blocking approvals, where execution pauses until approval is provided
- non-blocking approvals, where approval is deferred and the workflow resumes later

This is one of the most practical parts of the repo because many real systems cannot allow tools to execute autonomously without human review.

## 4A. Blocking Human Approval

Location: `human_in_the_loop/blocking_human_in_the_loop/`

### Files

- `blocking_example.py`: a minimal confirmation example around a fast-food order
- `hr_agent/agent.py`: a richer resumable HR workflow for reimbursements and time-off requests

### `blocking_example.py`

This is the simplest approval demo in the repo.

- `place_order(order_items)` places an order and calculates the total
- `confirmation_criteria(order_items)` requests approval only when the order total exceeds 100
- `FunctionTool(place_order, require_confirmation=confirmation_criteria)` wires the approval rule into the tool

The teaching point is that confirmation can be attached declaratively to a tool instead of being manually orchestrated in agent logic.

### `hr_agent/agent.py`

This demo shows a more explicit approval design:

- `reimburse(amount, tool_context)` always requires confirmation because it is wrapped with `FunctionTool(..., require_confirmation=True)`
- `request_time_off(days, tool_context)` auto-approves requests for 2 days or fewer
- for requests above 2 days, it calls `tool_context.request_confirmation(...)`
- once a confirmation payload is returned, the tool resumes and computes the approved number of days

The file also creates:

- a resumable `App`
- `ResumabilityConfig(is_resumable=True)`

That combination is what makes pause-and-resume behavior possible in a more realistic workflow.

### What This Section Teaches

- the difference between boolean confirmation and payload-based approval
- how tool execution can pause mid-flight
- why resumability matters when a human decision is not immediate

## 4B. Non-Blocking Human Approval

Location: `human_in_the_loop/non_blocking_human_in_the_loop/`

### Files

- `basic_human_in_loop_impl.py`: a simple in-memory demonstration
- `submit_requests.py`: persistent request submission and session continuation
- `approve_requests.py`: a separate approval console that resumes pending sessions
- `approval_requests.json`: local queue for pending approvals

### `basic_human_in_loop_impl.py`

This file demonstrates the mechanics of long-running tools in a single process:

- reimbursements under `$100` can be handled automatically by the agent logic
- larger reimbursements trigger `ask_for_approval(...)`
- `ask_for_approval` is wrapped as a `LongRunningFunctionTool`
- the script captures the generated function call and ticket id
- a simulated manager approval or rejection is entered via console input
- the agent is resumed by sending a `FunctionResponse` back into the same session

Run it with:

```bash
cd /Users/haruiz/open-source/agents_advanced_course/human_in_the_loop
uv run python non_blocking_human_in_the_loop/basic_human_in_loop_impl.py
```

### `submit_requests.py`

This is the more realistic version of the same idea.

It adds:

- persistent sessions using `DatabaseSessionService`
- a SQLite-backed `sessions.db`
- a JSON queue of pending approval requests in `approval_requests.json`
- an `after_agent_callback` that detects long-running tool events
- an interactive prompt so you can submit multiple reimbursement requests

When the agent emits a long-running tool call, the script stores:

- function call id
- tool name
- arguments
- user id
- session id
- pending status

That makes it possible to approve the request later from a different process.

Run it with:

```bash
cd /Users/haruiz/open-source/agents_advanced_course/human_in_the_loop/non_blocking_human_in_the_loop
uv run python submit_requests.py
```

### `approve_requests.py`

This companion script reads `approval_requests.json`, prompts the approver for each request, and resumes the correct session by sending a `FunctionResponse` back through the runner.

This is the key architectural pattern:

- one process submits work
- another process reviews queued approvals
- the original agent session resumes from persisted state

Run it with:

```bash
cd /Users/haruiz/open-source/agents_advanced_course/human_in_the_loop/non_blocking_human_in_the_loop
uv run python approve_requests.py
```

## Common Caveats

- model names in the repo are mixed and some are preview or older names; update them if your environment requires newer model IDs
- the repo expects environment variables to be present, but not every folder includes a complete setup guide
- the artifact demo assumes Pandoc is installed
- the SQL demo assumes a valid SQLite database path in `DB_PATH`
- the search tool in the paper-writing demo is referenced conceptually but not enabled in code

## Summary

This repo is best understood as a collection of teaching examples for agent engineering patterns:

- build an agent loop manually
- graduate to ADK orchestration
- produce artifacts instead of plain text
- insert humans into tool execution when autonomy should be constrained

If you want a single demo that is most representative of real-world workflows, start with `human_in_the_loop/non_blocking_human_in_the_loop/`. If you want the clearest conceptual starting point, start with `bulding_agent_from_scratch/`.
